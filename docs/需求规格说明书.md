## 1. 项目概述

### 1.1 项目名称
WebhookSender — Android 工作待办项快捷记录工具

### 1.2 项目背景
在运维、监控、通知等场景中，经常需要向 Webhook 地址推送格式化消息（如企业微信机器人、钉钉机器人、飞书机器人、自建通知服务等）。本 App 提供一个轻量级的移动端消息发送入口，方便用户随时随地发送结构化通知。主打"工作待办项"的快速记录与推送。

### 1.3 项目目标
- 提供一个简洁、易用的 Android 应用，用于工作待办项快捷记录
- 支持用户自定义 Webhook 地址
- 支持关键字+工作待办项+截止时间的消息组织结构
- 关键字记忆：发送后自动记住关键字，下次自动填充
- 记录发送历史并提供持久化存储（最多保留10条）
- 提供清晰的发送结果反馈

### 2.2 功能详细描述

#### F-01 Webhook 地址配置
- 用户通过一个设置对话框输入 Webhook URL
- URL 支持 HTTP 和 HTTPS 协议
- URL 保存到本地 DataStore 中
- App 启动时自动读取已保存的 URL
- 未配置 URL 时，发送界面提供醒目提示

#### F-02 待办项发送
- 主界面提供三个输入区域：
  - 优先级下拉框（P0高优 / P1一般 / P2低优，带颜色圆点标识）
  - 工作待办项文本框（多行输入，高度约 3-6 行）
  - 截止时间文本框（只读，点击弹出日期选择器）
- 点击发送按钮后，将数据组装为 JSON 格式（若截止时间不为空，text 字段自动拼接截止时间前缀）
- 以 POST 方式发送到配置的 Webhook 地址

#### F-02a 关键字记忆
- 发送成功后，自动将本次发送的"关键字"保存到 DataStore
- 下次打开 App 的 SendScreen 时，自动读取并填充到"关键字"输入框
- 用户可以手动修改关键字，修改后发送则更新为新的关键字
- Webhook 失败也视为一次发送，同样保存关键字

#### F-02b 截止时间选择
- 点击"截止时间"输入框，弹出 Material DatePicker 日期选择对话框
- 选择日期后格式化为 `yyyy-MM-dd` 显示在输入框中
- 发送时作为独立字段 `deadline` 随 JSON 发出
- 截止时间可选，不填则发送 JSON 中 deadline 为空字符串

#### F-03 发送结果反馈
- 发送期间按钮显示加载动画
- 发送成功后显示绿色成功提示
- 发送失败后显示红色错误提示
- 同时弹出 Toast 提示
- 错误原因包括：网络错误、HTTP 非 200、超时等

#### F-04 发送历史记录
- 每次发送（无论成功/失败）自动记录到本地数据库
- 记录字段：
  - 自增 ID
  - 关键字
  - 工作待办项内容（原"正文"）
  - 截止时间
  - 当前使用的 Webhook 地址
  - 发送时间戳
  - 是否成功
  - 错误摘要（失败时填写）

#### F-05 历史浏览
- 独立的历史页面，以列表形式展示
- 每条记录显示：
  - 关键字（加粗显示）
  - 工作待办项内容（最多两行，超出省略）
  - 截止时间（红色小字显示，若无则不显示）
  - 发送日期时间
  - 发送状态标签（绿色成功/红色失败）
- 列表按时间倒序排列
- 最多保留10条历史记录

#### F-11 意见反馈
- 独立底部导航页面**反馈**
- 输入框限制最多100字，`${content.length}/100` 显示当前字数
- 点击发送按钮后，将数据发送到固定的 DingTalk Webhook 地址
- 无论用户是否配置了自己的 Webhook，反馈页面均可独立使用
- 反馈内容自动以 `keyword="意见反馈"` 发送
- 发送结果和历史记录与待办消息统一存储，历史记录最多保存10条
- 反馈历史在历史卡片中以青色圆点标识，显示 `【意见反馈】` 前缀，不显示优先级和截止时间

## 3. 非功能需求

### 3.1 性能需求
| 项目 | 要求 |
|------|------|
| App 冷启动时间 | ≤ 2 秒 |
| 发送响应时间 | ≤ 15 秒（含网络延迟） |
| 历史记录加载 | 即时加载（最多10条） |

### 3.2 兼容性需求
- Android API 26（Android 8.0）及以上
- 自适应桌面模式和平板模式

### 3.3 安全性需求
- 仅允许 HTTPS 协议的网络通信（可调）
- Webhook 地址仅存储在本地
- 历史消息仅存储在本地数据库，不上传云端

### 3.4 可靠性需求
- 网络异常时优雅降级，记录失败状态
- 数据库操作失败时不影响 App 正常使用

## 5. 接口需求

### 5.1 出站接口（Webhook）

**请求方式**: POST  
**Content-Type**: application/json  

**请求体:**
```json
{
  "keyword": "string",
  "content": "string",
  "deadline": "2026-06-15",
  "text": "【截止时间：2026-06-15】\nkeyword\ncontent",
  "timestamp": 1717826400000
}
```

- 当 deadline 为空时，text = "keyword\ncontent"
- 当 deadline 不为空时，text 前缀自动拼接 `【截止时间：${deadline}】\n`

**超时配置:**
- 连接超时：10 秒
- 读取超时：15 秒

## 6. 数据需求

### 6.1 本地存储

| 存储内容 | 存储方案 | 说明 |
|----------|----------|------|
| Webhook URL | DataStore (SharedPreferences 现代化) | 键值对存储 |
| 上次关键字 | DataStore | 键值对存储，发送后自动保存 |
| 发送历史 | Room 数据库 | SQLite 持久化，最多10条 |

### 6.2 DataStore Schema

```
Key: "webhook_url" (stringPreferencesKey)
Value: 用户输入的 URL 字符串

Key: "last_keyword" (stringPreferencesKey)
Value: 上次发送的关键字字符串
```

### 6.3 数据库表结构

**表名**: `messages`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 自增主键 |
| keyword | TEXT | NOT NULL | 关键字（待办项通常为"todos"，反馈为"意见反馈"） |
| content | TEXT | NOT NULL | 内容（工作待办项或反馈意见） |
| priority | TEXT | NOT NULL DEFAULT '一般' | 优先级（P0高优/P1一般/P2低优），反馈为空 |
| deadline | TEXT | NOT NULL DEFAULT '' | 截止时间 (yyyy-MM-dd)，反馈为空 |
| webhookUrl | TEXT | NOT NULL | 发送时的 Webhook 地址 |
| sendTime | INTEGER | NOT NULL | 发送时间戳（毫秒） |
| success | INTEGER | NOT NULL | 0=失败, 1=成功 |
| errorSummary | TEXT | NULL | 失败时的错误简述 |

**历史清理策略**: 每次插入新记录后，执行 SQL 删除多余记录（不区分消息类型，待办和反馈统一计数）：
```sql
DELETE FROM messages 
WHERE id NOT IN (SELECT id FROM messages ORDER BY sendTime DESC LIMIT 10)
```

## 7. 限制与假设

- 用户目标 Webhook 服务端需要能接收标准 HTTP POST JSON 请求
- App 需要网络权限（INTERNET）
- 目标平台：Android 8.0+ (API 26+)
- 语言：中文（简体）UI
- 历史记录上限为10条，超出后旧记录自动移除

## 8. 验收标准

| 验收项 | 验收标准 |
|--------|----------|
| Webhook 配置 | 输入 URL → 保存 → 重启 App → 自动恢复 |
| 消息发送 | 填写内容 → 点击发送 → 服务端收到 POST 请求，格式正确 |
| 关键字记忆 | 发送时有关键字 → 退出 App → 重新打开 → 关键字自动填充 |
| 截止时间 | 点击截止时间 → 弹出日期选择器 → 选择日期 → 显示 yyyy-MM-dd |
| 成功反馈 | 发送成功后界面和 Toast 均提示成功 |
| 失败反馈 | 断网发送 → 界面和 Toast 均提示失败原因 |
| 历史记录 | 发送后切换到历史页 → 能看到刚发的记录（含截止时间） |
| 记录持久 | 杀掉 App 再打开 → 历史记录仍然存在 |
| 历史数量上限 | 发送超过10条 → 最早记录自动删除，仅保留最近10条 |
| 用途说明 | 首页顶部显示绿色提示卡片 |
