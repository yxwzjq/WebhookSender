**日期**: 2026-06-08

---

## 1. 项目概述

### 1.1 项目名称
WebhookSender — Android Webhook 消息发送工具

### 1.2 项目背景
在运维、监控、通知等场景中，经常需要向 Webhook 地址推送格式化消息（如企业微信机器人、钉钉机器人、飞书机器人、自建通知服务等）。本 App 提供一个轻量级的移动端消息发送入口，方便用户随时随地发送结构化通知。

### 1.3 项目目标
- 提供一个简洁、易用的 Android 应用
- 支持用户自定义 Webhook 地址
- 支持关键字+正文的消息组织结构
- 记录发送历史并提供持久化存储
- 提供清晰的发送结果反馈

## 2. 功能需求

### 2.1 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| F-01 | Webhook 地址配置 | 高 | 用户可输入并保存 Webhook URL |
| F-02 | 消息发送 | 高 | 填写关键字和正文后，向 Webhook 发送 POST 请求 |
| F-03 | 发送结果反馈 | 高 | 发送后显示成功/失败提示 |
| F-04 | 发送历史记录 | 高 | 记录每次发送的关键字、内容、时间、结果 |
| F-05 | 历史浏览 | 中 | 按时间倒序浏览所有发送记录 |
| F-06 | 历史持久化 | 高 | App 退出后历史记录不丢失 |
| F-07 | 设置持久化 | 高 | Webhook 地址保存后下次启动自动加载 |
| F-08 | 空状态提示 | 低 | 无历史记录时显示友好空状态 |

### 2.2 功能详细描述

#### F-01 Webhook 地址配置
- 用户通过一个设置对话框输入 Webhook URL
- URL 支持 HTTP 和 HTTPS 协议
- URL 保存到本地 DataStore 中
- App 启动时自动读取已保存的 URL
- 未配置 URL 时，发送界面提供醒目提示

#### F-02 消息发送
- 主界面提供两个文本输入框：
  - 关键字文本框（单行输入）
  - 正文文本框（多行输入，高度约 4 行）
- 点击发送按钮后，将数据组装为 JSON 格式：
```json
{
  "keyword": "警告",
  "content": "服务器CPU使用率95%",
  "text": "警告\n服务器CPU使用率95%",
  "timestamp": 1717826400000
}
```
- 以 POST 方式发送到配置的 Webhook 地址

#### F-03 发送结果反馈
- 发送期间按钮显示加载动画
- 发送成功后显示绿色成功提示
- 发送失败后显示红色错误提示
- 同时弹出 Toast 提示
- 错误原因包括：网络错误、HTTP 非 200、超时等

#### F-04 发送历史记录
- 每次发送（无论成功/失败）自动记录到本地数据库
- 记录字段：
  - 自增 ID
  - 关键字
  - 正文内容
  - 当前使用的 Webhook 地址
  - 发送时间戳
  - 是否成功
  - 错误摘要（失败时填写）

#### F-05 历史浏览
- 独立的历史页面，以列表形式展示
- 每条记录显示：
  - 关键字（加粗显示）
  - 正文内容（最多两行，超出省略）
  - 发送日期时间
  - 发送状态标签（绿色成功/红色失败）
- 列表按时间倒序排列

## 3. 非功能需求

### 3.1 性能需求
| 项目 | 要求 |
|------|------|
| App 冷启动时间 | ≤ 2 秒 |
| 发送响应时间 | ≤ 15 秒（含网络延迟） |
| 历史记录加载 | 即时加载（首屏 ≤ 100 条） |

### 3.2 兼容性需求
- Android API 26（Android 8.0）及以上
- 自适应桌面模式和平板模式

### 3.3 安全性需求
- 仅允许 HTTPS 协议的网络通信（可调）
- Webhook 地址仅存储在本地
- 历史消息仅存储在本地数据库，不上传云端

### 3.4 可靠性需求
- 网络异常时优雅降级，记录失败状态
- 数据库操作失败时不影响 App 正常使用

## 5. 接口需求

### 5.1 出站接口（Webhook）

**请求方式**: POST  
**Content-Type**: application/json  

**请求体:**
```json
{
  "keyword": "string",
  "content": "string",
  "text": "keyword\ncontent",
  "timestamp": 1717826400000
}
```

**超时配置:**
- 连接超时：10 秒
- 读取超时：15 秒

## 6. 数据需求

### 6.1 本地存储

| 存储内容 | 存储方案 | 说明 |
|----------|----------|------|
| Webhook URL | DataStore (SharedPreferences 现代化) | 键值对存储 |
| 发送历史 | Room 数据库 | SQLite 持久化 |

### 6.2 数据库表结构

**表名**: `messages`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 自增主键 |
| keyword | TEXT | NOT NULL | 关键字 |
| content | TEXT | NOT NULL | 正文内容 |
| webhookUrl | TEXT | NOT NULL | 发送时的 Webhook 地址 |
| sendTime | INTEGER | NOT NULL | 发送时间戳（毫秒） |
| success | INTEGER | NOT NULL | 0=失败, 1=成功 |
| errorSummary | TEXT | NULL | 失败时的错误简述 |

---

## 7. 限制与假设

- 用户目标 Webhook 服务端需要能接收标准 HTTP POST JSON 请求
- App 需要网络权限（INTERNET）
- 目标平台：Android 8.0+ (API 26+)
- 语言：中文（简体）UI

---

## 8. 验收标准

| 验收项 | 验收标准 |
|--------|----------|
| Webhook 配置 | 输入 URL → 保存 → 重启 App → 自动恢复 |
| 消息发送 | 填写内容 → 点击发送 → 服务端收到 POST 请求，格式正确 |
| 成功反馈 | 发送成功后界面和 Toast 均提示成功 |
| 失败反馈 | 断网发送 → 界面和 Toast 均提示失败原因 |
| 历史记录 | 发送后切换到历史页 → 能看到刚发的记录 |
| 记录持久 | 杀掉 App 再打开 → 历史记录仍然存在 |
